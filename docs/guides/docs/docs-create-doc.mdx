# 创建文档

创建一个 Markdown 文件 `greeting.md`，并将其放置在 `docs` 目录下。

```bash
website # 您网站的根目录
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: 创建一个丰富内容的文档页面。
---

# 来自 Docusaurus 的问候

您准备好为您的开源项目创建文档网站了吗？

## 标题

将显示在右上角的目录中

这样您的用户可以在不向下滚动或甚至不阅读太多内容的情况下了解此页面的内容。

## 默认情况下，只有 h2 和 h3 会出现在目录中。

您可以在每个文档中或在主题配置中配置目录的标题级别。

标题之间间隔良好，以便层次结构清晰。

- 列表将帮助您
- 呈现关键要点
- 您希望用户记住的内容
  - 您可以嵌套它们
    - 多次嵌套
```

:::note

在 `docs` 目录下，所有以下划线 (`_`) 开头的文件被视为“部分”页面，默认情况下将被忽略。

了解更多关于 [导入部分页面](../markdown-features/markdown-features-react.mdx#importing-markdown)。

:::

## 文档前端信息 {#doc-front-matter}

[前端信息](../markdown-features/markdown-features-intro.mdx#front-matter) 用于提供您文档页面的附加元数据。前端信息是可选的—Docusaurus 可以在没有前端信息的情况下推断出所有必要的元数据。例如，下面介绍的 [文档标签](#doc-tags) 功能需要使用前端信息。有关所有可能字段，请参见 [API 文档](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)。

## 文档标签 {#doc-tags}

标签在前端信息中声明，并在 [文档侧边栏](./sidebar/index.mdx) 之外引入另一种分类维度。

可以在行内定义标签，也可以引用在 [`tags file`](../../api/plugins/plugin-content-docs.mdx#tags-file) 中声明的预定义标签（可选，通常为 `docs/tags.yml`）。

在以下示例中：

- `docusaurus` 引用了在 `docs/tags.yml` 中声明的预定义标签键
- `Releases` 是一个行内标签，因为它在 `docs/tags.yml` 中不存在

```md title="docs/my-doc.md"
---
tags:
  - Releases
  - docusaurus
---

# 标题

内容
```

```yml title="docs/tags.yml"
docusaurus:
  label: 'Docusaurus'
  permalink: '/docusaurus'
  description: '与 Docusaurus 框架相关的文档'
```

:::tip

标签也可以使用 `tags: [Demo, Getting started]` 声明。

了解更多关于所有可能的 [Yaml 数组语法](https://www.w3schools.io/file/yaml-arrays/)。

:::

## 组织文件夹结构 {#organizing-folder-structure}

Markdown 文件在 `docs` 文件夹下的排列方式会对 Docusaurus 的内容生成产生多重影响。然而，其中大多数可以与文件结构解耦。

### 文档 ID {#document-id}

每个文档都有一个唯一的 `id`。默认情况下，文档的 `id` 是相对于根文档目录的文档名称（不带扩展名）。

例如，`greeting.md` 的 ID 是 `greeting`，而 `guide/hello.md` 的 ID 是 `guide/hello`。

```bash
website # 您网站的根目录
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

然而，`id` 的 **最后部分** 可以在前端信息中由用户定义。例如，如果 `guide/hello.md` 的内容如下定义，则其最终 `id` 为 `guide/part1`。

```md
---
id: part1
---

Lorem ipsum
```

ID 用于在手动编写侧边栏或使用与文档相关的布局组件或钩子时引用文档。

### 文档 URL {#doc-urls}

默认情况下，文档的 URL 位置是其相对于 `docs` 文件夹的文件路径，少数例外情况。即，如果文件名为以下之一，则文件名不会包含在 URL 中：

- 命名为 `index`（不区分大小写）：`docs/Guides/index.md`
- 命名为 `README`（不区分大小写）：`docs/Guides/README.mdx`
- 与父文件夹同名：`docs/Guides/Guides.md`

在所有情况下，默认的 slug 仅为 `/Guides`，而不包括 `/index`、`/README` 或重复的 `/Guides` 部分。

:::note

这个约定与 [类别索引约定](./sidebar/autogenerated.mdx#category-index-convention) 完全相同。然而，`isCategoryIndex` 配置不影响文档 URL。

:::

使用 `slug` 前端信息更改文档的 URL。

例如，假设您的网站结构如下：

```bash
website # 您网站的根目录
└── docs
    └── guide
        └── hello.md
```

默认情况下，`hello.md` 将在 `/docs/guide/hello` 处可用。您可以将其 URL 位置更改为 `/docs/bonjour`：

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug` 将附加到文档插件的 `routeBasePath`，默认值为 `/docs`。有关如何从 URL 中删除 `/docs` 部分，请参见 [仅文档模式](docs-introduction.mdx#docs-only-mode)。

:::note

可以使用：

- 绝对 slug：`slug: /mySlug`、`slug: /`...
- 相对 slug：`slug: mySlug`、`slug: ./../mySlug`...

:::

如果您希望文档在根目录可用，并且路径为 `https://docusaurus.io/docs/`，您可以使用 slug 前端信息：

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 侧边栏 {#sidebars}

在使用 [自动生成的侧边栏](./sidebar/autogenerated.mdx) 时，文件结构将决定侧边栏结构。

我们对文件系统组织的建议是：使您的文件系统镜像侧边栏结构（这样您就不需要手动编写 `sidebars.js` 文件），并使用 `slug` 前端信息自定义每个文档的 URL。
