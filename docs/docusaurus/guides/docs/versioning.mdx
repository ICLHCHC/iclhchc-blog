# 版本管理

您可以使用版本管理命令行工具根据 `docs` 目录中的最新内容创建新的文档版本。该特定的文档集将被保留并可访问，即使 `docs` 目录中的文档内容继续发展。

```mdx-code-block
import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';
```

:::warning

在开始为文档版本化之前，请仔细考虑——这可能会使贡献者很难协助改进文档！

:::

大多数情况下，您不需要进行版本管理，因为这会增加构建时间并给代码库带来复杂性。版本管理**最适合流量高且文档在版本之间变化迅速的网站**。如果您的文档很少变更，就不要为其添加版本管理。

要更好地理解版本管理的工作原理，并判断其是否符合您的需求，请继续阅读以下内容。

## 概述 {#overview}

一个典型的版本化文档网站结构如下所示：

```bash
website
├── sidebars.json        # 当前文档版本的侧边栏
├── docs                 # 当前文档版本的文档目录
│   ├── foo
│   │   └── bar.md       # https://mysite.com/docs/next/foo/bar
│   └── hello.md         # https://mysite.com/docs/next/hello
├── versions.json        # 指示可用版本的文件
├── versioned_docs
│   ├── version-1.1.0
│   │   ├── foo
│   │   │   └── bar.md   # https://mysite.com/docs/foo/bar
│   │   └── hello.md
│   └── version-1.0.0
│       ├── foo
│       │   └── bar.md   # https://mysite.com/docs/1.0.0/foo/bar
│       └── hello.md
├── versioned_sidebars
│   ├── version-1.1.0-sidebars.json
│   └── version-1.0.0-sidebars.json
├── docusaurus.config.js
└── package.json
```

`versions.json` 文件是按时间顺序排列的版本名称列表，从最新到最旧。

下表解释了版本化文件与其版本及生成的 URL 的对应关系。

| 路径                                    | 版本            | URL               |
| --------------------------------------- | -------------- | ----------------- |
| `versioned_docs/version-1.0.0/hello.md` | 1.0.0          | /docs/1.0.0/hello |
| `versioned_docs/version-1.1.0/hello.md` | 1.1.0 (最新)    | /docs/hello       |
| `docs/hello.md`                         | 当前版本       | /docs/next/hello  |

:::tip

`docs` 目录中的文件属于 `当前` 文档版本。

默认情况下，`当前` 文档版本标记为 `Next`，并托管在 `/docs/next/*` 下，但可以完全配置以适应您的项目发布生命周期。

:::

当前版本由**文件系统位置**定义，而最新版本由**导航行为**定义。它们可能是相同的版本，也可能不是！ （默认配置如上表所示，将其视为不同：当前版本在 `/docs/next`，最新版本在 `/docs`。）

## 教程 {#tutorials}

### 标记新版本 {#tagging-a-new-version}

1. 首先，确保当前文档版本（`./docs` 目录）准备好被冻结。
2. 输入新的版本号。

```bash npm2yarn
npm run docusaurus docs:version 1.1.0
```

在标记新版本时，文档版本管理机制将：

- 将 `docs/` 文件夹的完整内容复制到新建的 `versioned_docs/version-[versionName]/` 文件夹中。
- 创建基于您当前 [侧边栏](./sidebar/index.mdx) 配置的版本化侧边栏文件（如果存在） - 保存为 `versioned_sidebars/version-[versionName]-sidebars.json`。
- 将新版本号附加到 `versions.json`。

### 创建新文档 {#creating-new-docs}

1. 将新文件放入相应版本文件夹中。
2. 根据版本号在相应的侧边栏文件中包含对新文件的引用。

```mdx-code-block
<Tabs>
<TabItem value="当前版本结构">
```

```bash
# 新文件。
docs/new.md

# 编辑相应的侧边栏文件。
sidebars.js
```

```mdx-code-block
</TabItem>
<TabItem value="旧版本结构">
```

```bash
# 新文件。
versioned_docs/version-1.0.0/new.md

# 编辑相应的侧边栏文件。
versioned_sidebars/version-1.0.0-sidebars.json
```

```mdx-code-block
</TabItem>
</Tabs>
```

:::tip

版本化侧边栏文件与标准侧边栏文件一样，都是相对于给定版本的内容根目录——因此，对于上述示例，您的版本化侧边栏文件可能看起来像这样：

```json
{
  "sidebar": [
    {
      "type": "autogenerated",
      "dirName": "."
    }
  ]
}
```

或者用于手动侧边栏：

```json
{
  "sidebar": [
    {
      "type": "doc",
      "id": "new",
      "label": "新"
    }
  ]
}
```

:::

### 更新现有版本 {#updating-an-existing-version}

您可以同时更新多个文档版本，因为 `versioned_docs/` 中的每个目录代表发布时的特定路由。

1. 编辑任意文件。
2. 提交并推送更改。
3. 它将发布到相应版本。

示例：当您更改 `versioned_docs/version-2.6/` 中的任何文件时，它将仅影响版本 `2.6` 的文档。

### 删除现有版本 {#deleting-an-existing-version}

您也可以删除版本。

1. 从 `versions.json` 中移除该版本。

示例：

```diff
[
  "2.0.0",
  "1.9.0",
  // highlight-next-line
- "1.8.0"
]
```

2. 删除版本化文档目录。例如：`versioned_docs/version-1.8.0`。
3. 删除版本化侧边栏文件。例如：`versioned_sidebars/version-1.8.0-sidebars.json`。

## 配置版本管理行为 {#configuring-versioning-behavior}

“当前”版本是 `./docs` 文件夹的版本名称。管理版本有不同的方法，但两种非常常见的模式是：

- 您发布 v1，并立即开始在 v2 上工作（包括其文档）。在这种情况下，**当前版本**是 v2，位于 `./docs` 源文件夹中，可以在 `example.com/docs/next` 上浏览。**最新版本**是 v1，位于 `./versioned_docs/version-1` 源文件夹中，大多数用户在 `example.com/docs` 上浏览。
- 您发布 v1，并将在考虑 v2 之前维护它一段时间。在这种情况下，**当前版本**和**最新版本**都指向 v1，因为 v2 文档甚至还不存在！

Docusaurus 的默认设置非常适合第一种用例。我们将当前版本标记为“next”，您甚至可以选择不发布它。

**对于第二种用例**：如果您发布 v1 并且不打算在近期内着手 v2，那么与其为 v1 进行版本化并必须在两个文件夹（`./docs` + `./versioned_docs/version-1.0.0`）中维护文档，不如考虑“假装”当前版本是一个切分版本，通过给予它路径和标签：

```js title="docusaurus.config.js"
export default {
  presets: [
    '@docusaurus/preset-classic',
    docs: {
      // highlight-start
      lastVersion: 'current',
      versions: {
        current: {
          label: '1.0.0',
          path: '1.0.0',
        },
      },
      // highlight-end
    },
  ],
};
```

`./docs` 中的文档将服务于 `/docs/1.0.0` 而不是 `/docs/next`，`1.0.0` 将成为我们在导航栏下拉菜单中链接的

默认版本，您只需维护一个 `./docs` 文件夹。

我们提供这些插件选项来自定义版本管理行为：

- `disableVersioning`：即使存在版本也明确禁用版本管理。这将使网站仅包含当前版本。
- `includeCurrentVersion`：包含文档的当前版本（`./docs` 文件夹）。
  - **提示**：如果当前版本是一个正在进行的项目，尚未准备好发布，请关闭此选项。
- `lastVersion`：设置“最新版本”（`/docs` 路由）所指向的版本。
  - **提示**：如果您的当前版本指的是一个不断修补和发布的主要版本，则 `lastVersion: 'current'` 是合适的。最新版本的实际路由基本路径和标签是可配置的。
- `onlyIncludeVersions`：定义从 `versions.json` 部署的版本子集。
  - **提示**：在开发和部署预览中限制到 2 或 3 个版本，以提高启动和构建时间。
- `versions`：版本元数据的字典。对于每个版本，您可以自定义以下内容：
  - `label`：在版本下拉菜单和横幅中显示的标签。
  - `path`：此版本的路由基本路径。默认情况下，最新版本为 `/`，当前版本为 `/next`。
  - `banner`：`'none'`、`'unreleased'` 和 `'unmaintained'` 之一。决定在每个文档页面顶部显示的内容。任何版本高于最新版本的将被标记为“未发布”，低于的将被标记为“未维护”。
  - `badge`：在该版本的文档顶部显示版本名称的徽章。
  - `className`：为该版本文档页面的 `<html>` 元素添加自定义 `className`。

有关更多详细信息，请参阅 [文档插件配置](../../api/plugins/plugin-content-docs.mdx#configuration)。

## 导航栏项 {#navbar-items}

我们提供多个导航栏项，帮助您快速设置导航，而不必担心版本化路由。

- [`doc`](../../api/themes/theme-configuration.mdx#navbar-doc-link)：链接到文档。
- [`docSidebar`](../../api/themes/theme-configuration.mdx#navbar-doc-sidebar)：链接到侧边栏中的第一个项。
- [`docsVersion`](../../api/themes/theme-configuration.mdx#navbar-docs-version)：链接到当前查看版本的主要文档。
- [`docsVersionDropdown`](../../api/themes/theme-configuration.mdx#navbar-docs-version-dropdown)：包含所有可用版本的下拉菜单。

这些链接都会按照以下顺序查找适当的版本进行链接：

1. **活动版本**：用户当前正在浏览的版本，如果她在由此文档插件提供的页面上。如果她不在文档页面上，则回退到...
2. **首选版本**：用户最后查看的版本。如果没有历史记录，则回退到...
3. **最新版本**：我们导航到的默认版本，由 `lastVersion` 选项配置。

## 推荐实践 {#recommended-practices}

### 仅在需要时对文档进行版本管理 {#version-your-documentation-only-when-needed}

例如，您正在为 npm 包 `foo` 构建文档，当前版本为 1.0.0。然后您发布了一个补丁版本，修复了一个小错误，现在版本为 1.0.1。

您是否应该为新版本 1.0.1 切分新的文档版本？**您可能不应该**。根据语义版本控制，1.0.1 和 1.0.0 的文档不应有差异，因为没有新功能！为此切分新版本只会产生不必要的重复文件。

### 保持版本数量较小 {#keep-the-number-of-versions-small}

作为一个好的经验法则，尝试将您的版本数量保持在 10 个以下。您**很可能**会拥有许多过时的版本文档，几乎没有人再阅读它们。例如，[Jest](https://jestjs.io/versions) 目前是版本 `27.4`，仅维护几个最新的文档版本，最低为 `25.X`。保持数量小 😊

:::tip 归档旧版本

如果您在 Jamstack 提供商（例如 [Netlify](../../deployment.mdx)）上部署您的网站，提供商将把每个生产构建作为快照保存在不可变 URL 下。您可以将归档版本作为这些不可变 URL 的外部链接包含在内。Jest 网站和 Docusaurus 网站都使用这种模式以保持活动构建版本的数量较少。

:::

### 在文档中使用绝对导入 {#use-absolute-import-within-the-docs}

不要在文档中使用相对路径导入。因为当我们切分版本时，路径将不再有效（嵌套级别不同，以及其他原因）。您可以使用 Docusaurus 提供的 `@site` 别名，它指向 `website` 目录。例如：

```diff
- import Foo from '../src/components/Foo';
+ import Foo from '@site/src/components/Foo';
```

### 按文件路径链接文档 {#link-docs-by-file-paths}

通过 `.md` 扩展名的相对文件路径引用其他文档，以便 Docusaurus 在构建时可以将其重写为实际的 URL 路径。文件将链接到正确对应的版本。

```md
[@hello](hello.mdx#paginate) 文档很不错！

请参阅 [教程](../getting-started/tutorial.mdx) 获取更多信息。
```

### 全局或版本化的协作资产 {#global-or-versioned-collocated-assets}

您应决定资产（如图片和文件）是按版本分配还是在不同版本间共享。

如果您的资产应该版本化，请将其放入文档版本中，并使用相对路径：

```md
![图像替代文本](./myImage.png)

[下载此文件](./file.pdf)
```

如果您的资产是全局的，请放入 `/static` 中并使用绝对路径：

```md
![图像替代文本](/myImage.png)

[下载此文件](/file.pdf)
```
